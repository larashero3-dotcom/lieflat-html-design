Thesis

核心变化：模型开始承担“过程”。

过去的模型更像高质量函数调用：你拆任务、控步骤、校验结果。GPT-5.5 的差异在于它更早理解目标，更少请求引导，更稳定地规划、用工具、检查并继续。

thesis02

What changed

四个能力区，构成 GPT-5.5 的实际边界。

这些能力的实际边界，是编码修复、电脑操作、知识工作和提示词瘦身同时进入同一轮任务。

Long context

把需求、代码、日志、文档放进同一个工作面，而不是反复摘要。

Tool use

更适合多步工具链：查、写、跑、验证、再修正。

Coding depth

从补全片段转向理解代码库、拆任务、处理回归风险。

Reasoning budget

按任务复杂度分配思考强度，避免每次都用同一种成本结构。

Agentic coding

适合从 issue 到 patch 的闭环：定位、修改、跑测试、解释风险。

Grounded assistant

适合依赖企业知识库的问答、总结、合规校验和多源综合。

03

Spec to plan

适合把产品规格转成任务树、依赖关系、验收条件和执行顺序。

Use cases

优先切入“多步、可验证、高摩擦”的工作。

不要先迁移所有 prompt；先选择一个“人类一直在补锅”的流程。

use-case fit04

Product strategy

GPT-5.5 把“模型选择”变成运营问题。

用 eval 做路由，而不是用“最强模型”做默认答案：低延迟任务走低成本路线，高价值复杂任务才进入深度推理。

01

Show progress

让用户看到模型当前在读什么、改什么、等什么。

02

Give control

把审批、回滚、重试和约束做成产品控制，而不是隐藏在提示词里。

03

Verify work

把测试、引用、日志和差异检查设计成默认动作。

Operating risks

能力越像执行者，治理就越要像系统工程。

越会用工具，越需要清楚的工具描述、参数约束、证据规则、确认点和审计日志。

1

权限边界

把读、写、发布、删除、支付等动作拆成不同授权层。

2

可观察性

记录模型看过的上下文、调用的工具、关键判断和失败分支。

3

验收机制

让模型输出可检查的证据，而不是只输出自信的结论。

risk controls07

Bottom line

不是更大的聊天框，而是更小的管理负担。

GPT-5.5 的产品问题不是“它能不能回答”，而是“它能不能在你少操心的情况下，把复杂工作推进到可验收状态”。
